# Session summary — 2026-05-17

A long debugging session that ended up publishing this repo. What worked, what didn't, what's still open.

## What works

- **Boot the nexmon firmware on Pi 4 + kernel 6.12** (D10 minimal CSI build only — the full build firmware-traps).
- **Modified `brcmfmac` driver** ported from nexmon 6.6.y to kernel 6.12 API. Built and loads cleanly.
- **`nexutil` with `USE_VENDOR_CMD=1`**. Sends IOCTLs to the patched firmware. Confirmed working with reads (`-k` for chanspec, `-o` for objmem).
- **cfg80211 monitor mode** on `wlan0` (`iw dev wlan0 set type monitor`) combined with nexmon monitor mode (`nexutil -Iwlan0 -m1`). Both are required.
- **Live 802.11 capture** via `tcpdump -i wlan0`. Verified by parsing 466 frames from an 8-second capture and pulling 60+ beacons across 20+ access points.
- **Object memory READ** from userspace via `nexutil -Iwlan0 -o <byte_addr> -l <bytes>`. Byte-addressed (nexmon source uses word indices, multiply by 2).

## What doesn't work yet

- **CSI subcarriers in the canonical `nexmon_csi` UDP/5500 format**. Requires `process_frame_hook` in firmware, which crashes init on Pi 4 + kernel 6.12. See `findings/2026-05-17-d10-minimal-csi-recipe.md`.
- **Configuring CSI via IOCTL 500** has two failure modes: (a) sometimes triggers a firmware trap, (b) when it doesn't trap, the SHM writes silently no-op because the IOCTL handler is gated on `wlc->hw->up` which is FALSE in pure monitor mode. Tried "associate first, then send IOCTL 500, then switch to monitor" — IOCTL returned rc=0 but the target SHM didn't change. See `findings/2026-05-17-shm-access-via-nexutil.md`.
- **SHM WRITE from userspace** — no existing IOCTL handler exposes raw SHM write. Would need to add `NEX_WRITE_OBJMEM` to `nexmon_csi/src/ioctl.c` and rebuild firmware (which itself is the hard part).

## Things we tried that didn't help

- Disabling individual hooks one at a time (D1, D2, D3, D4)
- Disabling the UCODE patch (D5)
- Building a fresh full firmware (D6)
- Blacklisting `brcmfmac_wcc` (D7)
- Building the modified `brcmfmac` driver and using it with the full firmware (D8 — drives load fine but firmware still traps)
- Stub `process_frame_hook` returning immediately (D11)
- Adding `str r5, [sp, #8]` preservation to the prehook asm (D12)
- Real RX path body with range/null guards (D13b)
- Trampoline preserving original instructions verbatim (D14, D18)
- Sending IOCTL 500 in associated managed mode then switching to monitor

The deep bisection result is that the **mere presence of any C function in nexmon's patch RAM (0x22A95C - 0x22E95C)** interacts with how the BCM43455c0 firmware uses that region during init. The D10 minimal build (only `__attribute__((at()))` patches, no C function bodies) is the only nexmon_csi-flavored build that boots.

## Open paths forward

1. **Patch the IOCTL handler to skip the `hw->up` gate** for SHM-write paths. If the chip is initialized enough to accept SHM writes (even in `hw->up=0` monitor-only state), this would unlock IOCTL 500 from monitor mode. Risk: writes might fail or hang chip if `hw->up=0` for real reasons.
2. **Add an `NEX_WRITE_OBJMEM` IOCTL handler** to `ioctl.c` and rebuild firmware. Lets userspace write arbitrary SHM. Same `hw->up` issue.
3. **Hook a different firmware function entry**, not the loop-internal address at `0x1C1C3E`. Specifically: find a function that's ONLY called at real RX time (not during init), hook there. Requires deeper firmware RE.
4. **Userspace CSI parser from monitor captures**. The chip with D10 firmware still has the ucode patches active and writes CSI markers somewhere. They flow through `wlc_recv` and into the monitor interface. Parse the monitor pcap, find the marker frames (RxFrameSize==2 in the doubled d11rxhdr), and reconstruct subcarriers. Possible but requires reverse-engineering the marker frame layout.
5. **Use the Pi 5 Discussion #395 procedure** — confirmed working there. Not applicable to Pi 4 (we tried). But if hardware budget allows, a Pi 5 with this same userspace tooling would Just Work.

## Recommendations for next session

If you keep iterating on this firmware path, the highest-leverage thing is option 4 (userspace parser from monitor captures). It doesn't require firmware modification, doesn't risk bricking the Pi, and reuses the working D10 stack. The CSI data IS being generated by the ucode — we just don't have code that exports it in the standard format. Writing that code in Python operating on tcpdump output keeps the firmware untouched.

If you want full canonical UDP/5500 CSI on the Pi 4, the realistic options are: (a) Pi OS Bullseye + kernel 5.15 (legacy modified driver works), (b) Pi 5 upgrade, or (c) ESP32-C6 distributed sensors as a complementary capture path.
